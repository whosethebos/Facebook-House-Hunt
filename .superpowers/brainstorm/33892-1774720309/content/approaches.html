Architecture Approaches

Three ways to structure the Facebook scraping + agent pipeline

A

Single Orchestrator + Playwright Scraper

One Google ADK agent orchestrates everything: discovers groups, scrapes posts page-by-page using Playwright, feeds posts to Ollama for analysis, streams results via SSE. Simple, linear pipeline — same pattern as Amazon-Purchase.

Pros

  • Simple to build and debug
  • Familiar pattern (matches Amazon project)
  • Easy to stream progress

Cons

  • Sequential — groups scraped one at a time
  • Single point of failure
B

Multi-Agent: Scraper + Analyst + Ranker (Recommended)

ADK orchestrator coordinates 3 sub-agents: GroupDiscoveryAgent finds relevant Facebook groups, ScraperAgent extracts posts + images, AnalystAgent uses Ollama to score each post against criteria and rank results. All agents report progress via SSE.

Pros

  • Clean separation of concerns
  • Easy to extend (add new agents)
  • Each agent independently testable
  • Matches ADK's multi-agent philosophy

Cons

  • More moving parts
  • Slightly more setup
C

Multi-Agent + Parallel Scraping

Same as B, but the ScraperAgent spawns parallel Playwright sessions to scrape multiple Facebook groups simultaneously. Faster results, but higher memory use and higher risk of Facebook detecting bot behavior.

Pros

  • Significantly faster for many groups
  • Results appear sooner in UI

Cons

  • Higher Facebook bot detection risk
  • Higher memory/CPU on local machine
  • More complex to debug
